# KIP Blueprint 审稿指南

本指南面向参与 KIP (Kervaire Invariant Problem) 形式化项目的数学家，介绍蓝图系统的设计逻辑、依赖图的阅读方式，以及如何在 GitHub Pages 上对蓝图节点进行审核。

---

## 1. 蓝图是什么

蓝图 (Blueprint) 把论文中的数学内容拆成一个个独立的**节点** (node)，每个节点是一条数学声明 (definition / theorem / axiom 等)。节点之间通过 `\uses{...}` 标注建立依赖关系，形成一张有向无环的**依赖图**。

蓝图的作用不是展示，而是**调度**：

- 哪些结果已经足够稳定，可以开始形式化？
- 哪些结果卡在同一个瓶颈上？
- 哪些任务可以并行推进？
- 主定理还缺哪些上游节点？

> 蓝图托管在 GitHub Pages: `https://surenny.github.io/KIP/blueprint/`

---

## 2. 三类节点

蓝图中的数学声明被分为三类，在依赖图中用不同的**形状**区分：

| 形状 | 类型 | 包含的 LaTeX 环境 | 说明 |
|:---:|:---:|:---|:---|
| □ 方形 | **定义 (Definition)** | `definition` | 引入新概念，没有"证明"这一说 |
| ○ 椭圆 | **定理 (Theorem)** | `theorem`, `lemma`, `proposition`, `corollary`, `remark` | 需要证明的命题 |
| ◇ 菱形 | **公理 (Axiom)** | `axiom` | 外部假设或未形式化的已知结果，不需要在本项目内证明 |

分类的意义在于：定义只需要"陈述"就算完成；定理需要走完整的证明流程；公理则标记了我们与外部数学知识的边界。

---

## 3. 节点的生命周期

每个节点从创建到完成，经过以下几个阶段。不同阶段在依赖图中用**边框颜色**表示：

```
Draft ──→ Reviewed ──→ Bound ──→ Aligned ──→ Proved
(草稿)     (已审核)     (已绑定)    (已对齐)    (已证明)
```

### 各阶段详解

| 阶段 | 边框颜色 | 填充颜色 | 含义 | 谁来做 |
|:---|:---:|:---:|:---|:---|
| **Draft** (草稿) | 灰色 `#999` | 无 | 节点刚被创建，自然语言陈述尚未审核 | AI |
| **Reviewed** (已审核) | 橙色 `#E69F00` | 无 | 数学家确认自然语言陈述是准确的 | 数学家 |
| **Bound** (已绑定) | 蓝色 `#0072B2` | 无 | 节点已通过 `\lean{...}` 绑定到 Lean 声明 | AI |
| **Aligned** (已对齐) | 绿色 `#009E73` | 浅绿色 `#D4EFDF` | 数学家确认 Lean 声明与自然语言陈述一致 |人类|
| **Proved** (已证明) | 深绿色 `#1CAC78` | 深绿色 `#1CAC78` | Lean 证明已通过编译 (仅限定理类节点) |AI |

### 不同类型节点的完成标准

- **定义 (□)**：到达 **Aligned** 即为完成 — 定义没有"证明"
- **定理 (○)**：到达 **Proved** 才算完成 — 需要 Lean 证明通过编译
- **公理 (◇)**：到达 **Aligned** 即为完成 — 公理不需要证明

### NL 内容变更检测

系统会自动跟踪每个节点的自然语言内容哈希值。如果作者修改了某个节点的 LaTeX 文本，系统会自动重置该节点的 Reviewed 和 Aligned 状态，要求重新审核。你不需要手动关注这件事——如果一个你之前审核过的节点突然变回了 Draft 状态，说明它的内容已被作者修改。

## 4. 依赖图的阅读方式

### 章节对应关系

依赖图按照蓝图的**章节**组织。每个章节有自己的依赖图页面，当前的章节包括：

| 章节 | 内容 |
|:---|:---|
| Prerequisites | 预备知识：谱、同伦群、Adams 谱序列、合成谱等 |
| Synthetic Spectra | 合成谱的基本构造和性质 |
| Synthetic Extensions | 合成扩张谱序列 |
| Extension Spectral Sequence | 扩张谱序列 |
| Extensions on E_r page | E_r 页上的扩张 |
| Leibniz-Mahowald | Leibniz-Mahowald 定理 |
| Proof of Main Theorem | 主定理的证明 |

你可以从蓝图首页点击进入每个章节，然后通过该章节的依赖图页面查看节点之间的关系。

### 如何看图

- **箭头方向**：A → B 表示"B 的证明依赖 A"
- **节点颜色**：反映当前所处的生命周期阶段（见上表）
- **点击节点**：弹出详情面板，可以查看自然语言陈述、Lean 代码（如果已绑定）、评论，以及进行审核操作

## 5. 如何在 GitHub Pages 上审核

蓝图部署在 GitHub Pages 上后，你可以直接在浏览器中进行审核操作，无需搭建本地环境。

### 5.1 打开蓝图

访问 `https://surenny.github.io/KIP/blueprint/`，进入你要审核的章节的依赖图页面。

### 5.2 点击节点

点击依赖图中的任意节点，会弹出一个详情面板，包含：

- 节点的自然语言陈述
- Lean 声明（如果已绑定）
- 当前状态（Draft / Reviewed / Bound / Aligned / Proved）
- 审核操作按钮

### 5.3 审核操作

在节点详情面板中，你可以进行以下操作：

#### Approve NL (审核自然语言)

- **按钮**：橙色的 **"Approve NL"**
- **作用**：确认节点的自然语言陈述是准确、完整的
- **什么时候点**：当你读完陈述，认为它正确地描述了数学内容
- **状态变化**：Draft → Reviewed

#### Confirm Alignment (确认对齐)

- **按钮**：绿色的 **"Confirm Alignment"**
- **作用**：确认 Lean 声明与自然语言陈述一致
- **什么时候点**：当节点已有 Lean 绑定 (Bound)，且你确认两者表达的是同一个数学内容
- **状态变化**：Bound → Aligned
- **前提**：节点必须已经处于 Bound 状态

#### 发表评论

- **话题选择**：`general`（一般评论）、`review`（NL 审核相关）、`align`（对齐相关）
- **文本框**：输入你的评论内容
- **按钮**：灰色的 **"Send"**

#### 建议重分类

- **下拉菜单**：选择 `definition`、`theorem` 或 `axiom`
- **按钮**：粉色的 **"Suggest"**
- **作用**：如果你认为某个节点的分类不对（比如一个 axiom 实际上应该是 theorem），可以建议修改

#### 填写审核人

- 面板中有一个**审核人姓名**输入框，请填写你的名字。系统会记住你的名字，下次自动填充。

### 5.4 提交审核

在 GitHub Pages 模式下，你的所有审核操作**不会立即生效**，而是暂存在浏览器的 localStorage 中。

页面右下角会出现一个浮动工具栏：

```
┌─────────────────────────────────────┐
│  📋 N pending change(s)  [Commit Reviews]  ⚙  │
└─────────────────────────────────────┘
```

- **N pending change(s)**：显示你有多少条待提交的审核
- **Commit Reviews**：点击后，所有待提交的审核会合并为**一次 Git commit**，写入仓库的 `blueprint/status.yaml`
- **⚙ (设置)**：打开设置面板

### 5.5 首次使用：配置 GitHub PAT

第一次点击 "Commit Reviews" 时，系统会要求你提供一个 GitHub Personal Access Token (PAT)。这是因为 GitHub Pages 是静态页面，需要通过 GitHub API 才能向仓库写入数据。

**如何创建 PAT：**

1. 打开 [GitHub Token 设置页面](https://github.com/settings/tokens?type=beta)
2. 点击 **"Generate new token"**
3. 填写 Token 名称，例如 `KIP Blueprint Review`
4. **Repository access** 选择 **"Only select repositories"**，然后选中 `surenny/KIP`
5. **Permissions** → **Repository permissions** → **Contents** 设置为 **Read and write**
6. 点击 **Generate token**，复制生成的 token

然后在蓝图页面的设置面板（⚙）中粘贴 token。

> **安全说明**：Token 仅存储在你的浏览器 localStorage 中，不会被发送到任何第三方服务器。页面只会在你点击 "Commit Reviews" 时，通过 GitHub API 直接与 GitHub 通信。

### 5.6 设置面板

点击浮动工具栏的 ⚙ 按钮，打开设置面板：

- **Reviewer Name**：你的审核人姓名（会预填到所有审核操作中）
- **GitHub PAT**：你的 Personal Access Token
- **Save**：保存设置
- **Clear Pending**：清除所有未提交的审核（如果你想放弃当前的修改）

## 6. 审核工作流建议

### 推荐流程

1. **按章节审核**：选择一个章节，从依赖图的叶节点（没有出边的节点）开始，逐步向上游审核
2. **先看定义**：确保定义准确无歧义
3. **再看定理陈述**：确认定理陈述与论文原文一致
4. **最后看依赖**：确认 `\uses` 标注的依赖关系是正确的（如果发现缺失或错误的依赖，请通过评论指出）
5. **一次性提交**：完成一个章节的审核后，点击 "Commit Reviews" 统一提交

### 审核要点

审核 NL (自然语言) 时，请关注：

- 陈述是否数学上准确？
- 是否遗漏了重要的条件或假设？
- 符号使用是否一致？
- 分类是否正确？（是定义、定理还是公理？）

审核 Alignment (对齐) 时，请关注：

- Lean 声明的类型签名是否与自然语言陈述匹配？
- 是否有信息在形式化过程中丢失？
- 泛化程度是否合适？（既不过度泛化也不过度特化）

### 遇到问题时

- 如果某个节点的陈述有问题，**不要点 Approve NL**，而是通过评论功能说明问题
- 如果 Lean 声明与自然语言不一致，**不要点 Confirm Alignment**，而是添加评论
- 如果提交时遇到 "Conflict" 错误，说明有人在你之前提交了修改——请刷新页面后重试

## 7. 图例速查

以下是依赖图中你会看到的所有视觉元素：

### 形状

```
 ┌───┐
 │   │  方形 = 定义 (Definition)
 └───┘

 (   )  椭圆 = 定理 (Theorem / Lemma / Proposition / Corollary)

  / \
 /   \  菱形 = 公理 (Axiom)
 \   /
  \ /
```

### 颜色

```
 ■ 灰色 (#999999)     Draft    — 草稿，尚未审核
 ■ 橙色 (#E69F00)     Reviewed — 自然语言已审核
 ■ 蓝色 (#0072B2)     Bound    — 已绑定 Lean 声明
 ■ 绿色 (#009E73)     Aligned  — 已确认 NL ↔ Lean 对齐
 ■ 深绿 (#1CAC78)     Proved   — Lean 证明已通过（仅定理）
```

### 边框 vs 填充

- **边框颜色**：表示节点当前所处的生命周期阶段
- **填充颜色**：
  - 无填充 = Draft / Reviewed / Bound
  - 浅绿色填充 = Aligned
  - 深绿色填充 = Proved

## 8. 常见问题

**Q: 我不懂 Lean，能参与审核吗？**

当然可以。审核自然语言 (Approve NL) 完全不需要 Lean 知识——你只需要判断数学陈述本身是否准确。Confirm Alignment 需要对 Lean 声明有基本的理解，但如果你不确定，可以跳过这一步。

**Q: 我的审核会影响其他人吗？**

会。你的审核状态会写入仓库的 `status.yaml` 文件，其他人刷新页面后会看到更新后的节点颜色。如果作者后来修改了节点内容，系统会自动重置该节点的审核状态。

**Q: 我能撤回审核吗？**

在点击 "Commit Reviews" 之前，你可以通过设置面板的 "Clear Pending" 清除所有未提交的操作。提交之后，需要联系项目维护者手动修改 `status.yaml`。

**Q: PAT 过期了怎么办？**

重新生成一个新的 PAT，然后在设置面板（⚙）中更新即可。

**Q: 为什么某个节点之前是 Reviewed，现在变回 Draft 了？**

这说明作者修改了该节点的 LaTeX 文本内容。系统检测到文本变更后，会自动重置审核状态，需要重新审核。
